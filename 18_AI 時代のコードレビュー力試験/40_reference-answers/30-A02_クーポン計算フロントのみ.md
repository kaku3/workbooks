# A-02: クーポン計算をサーバーに移す — 模範解答

## 設問 1：現状の処理フローと問題点

**現状のフロー**：

1. `checkout.html` が `GET /api/coupons/:code` でクーポン情報を取得
2. フロントエンドで割引額を計算（`discountRate` を使って `Math.floor(total * discountRate / 100)`）
3. `POST /api/orders` に送るリクエストボディは `{ address: address }` のみ → **クーポン情報・割引情報はサーバーに送っていない**
4. サーバーは受け取った金額を検証せずに注文を作成

**問題点**：

- クーポン割引が「フロントエンドのみ」で計算・表示されており、サーバー側での検証がない
- ブラウザの開発者ツールで `POST /api/orders` のリクエストを改ざんすれば、任意の金額で注文を確定できる
- クーポンコードの有効性（期限切れ・使用済み・不正なコード）もサーバーで検証されていない

> **採点ポイント**：「フロントだけで計算している」だけでなく「リクエストに金額が含まれていない」→「サーバーが検証できない」という流れを説明できているかを見る。

## 設問 2：金額改ざんの具体的方法

ブラウザの開発者ツールを使って：

1. ネットワークタブで `POST /api/orders` のリクエストをインターセプト（または JavaScript で直接 fetch を呼ぶ）
2. リクエストボディを `{ address: "東京都", totalAmount: 1 }` のように金額を書き換えて送信
3. サーバーは受け取った値をそのまま注文として保存する

また、クーポンについては：
- 無効なクーポンコードを自分で試せば「どんな文字列でも通るか」を試せる
- `GET /api/coupons/INVALID` を叩いて 404 なら弾かれているが、現状 `app.js` のクーポン API は `COUPONS` 配列を検索するだけで、使用済みチェックがない

> **採点ポイント**：具体的な攻撃手順を説明できるかを見る。「改ざんできる」と言うだけでは △。

## 設問 3：サーバーサイドクーポン検証の設計

修正方針：`POST /api/orders` でクーポンコードを受け取り、サーバー側で再計算・検証する。

```js
// POST /api/orders のリクエストボディ（修正後）
// { address: "東京都", productId: 1, quantity: 1, couponCode: "SAVE10" }

app.post('/api/orders', (req, res) => {
  const { address, productId, quantity, couponCode } = req.body

  // 商品価格をサーバーで取得
  const products = JSON.parse(fs.readFileSync('./db/products.json'))
  const product = products.find(p => p.id === productId)
  let total = product.price * quantity

  // クーポン検証（サーバー側）
  if (couponCode) {
    const coupon = COUPONS.find(c => c.code === couponCode)
    if (!coupon) return res.status(400).json({ error: '無効なクーポンコードです' })
    total = Math.floor(total * (1 - coupon.discountRate / 100))
  }

  // 税計算・注文保存（以降は既存処理）
})
```

**修正が必要なファイル**：
- `app.js`：クーポン検証・金額計算をサーバー側に移動
- `checkout.html`：クーポンコードをリクエストに含めて送る。フロントの計算はあくまで「表示用プレビュー」とする

> **採点ポイント**：「サーバー側で再計算する」方針と、フロントは表示用・サーバーが正の計算を持つという役割分担を説明できているかを見る。

## 設問 4：フロントでプレビュー表示する理由と注意点

ユーザー体験のために、サーバーに送る前にフロントでプレビュー計算することは合理的。

**理由**：
- 「注文確定」ボタンを押す前に割引後の金額をユーザーに見せるため
- サーバーラウンドトリップなしに即座に表示できる

**注意点**：
- フロントの計算はあくまで「表示用」であり、実際の決済額はサーバー側が決定する
- フロントとサーバーの計算ロジックを一致させておかないと「表示と実際の請求額が違う」という問題が起きる
- フロント計算結果をサーバーに「答え」として送ってはいけない。サーバーは常に自分で計算する

> **採点ポイント**：フロントプレビューの存在意義を否定せず、「検証はサーバーが行う」という原則を明示できているかを見る。
