# M-02: トークンの実装が2種類ある — 模範解答

## 設問 1：2種類のトークン生成の差

**`app.js` の実装**（ログイン API 内）：
```js
const token = Buffer.from(String(user.id)).toString('base64')
// 例：id=1001 → "MTAwMQ=="
```

**`utils.js` の実装**（`generateToken` 関数）：
```js
function generateToken(userId) {
  return Buffer.from(userId + ':' + Date.now()).toString('base64')
}
// 例：id=1001, timestamp=1700000000000 → "MTAwMToxNzAwMDAwMDAwMDAw"
```

差異：
| 項目 | `app.js` | `utils.js` |
|------|---------|-----------|
| フォーマット | `{userId}` のみ | `{userId}:{timestamp}` |
| 逆変換 | `atob(token)` でIDが得られる | `:` で split が必要 |
| 互換性 | ❌ 両者のトークンに互換性なし | ❌ |

> **採点ポイント**：「2種類ある」だけでなく「どう違うか」を具体的に示せているかを見る。

## 設問 2：`getUserIdFromToken` が偶然動く理由

`utils.js` の `getUserIdFromToken` は：

```js
function getUserIdFromToken(token) {
  const decoded = Buffer.from(token, 'base64').toString()
  return decoded.split(':')[0]
}
```

`app.js` が生成するトークンは `MTAwMQ==`（Base64 の `1001`）なので、デコードすると `"1001"` になる。

`"1001".split(':')[0]` → `"1001"` → **正しいユーザーIDが返ってくる**

ただし、これは `:` を含まない `app.js` 形式のトークンでたまたま動いているだけで、`utils.js` が想定している形式と互換性があるわけではない。

→ `utils.js` が生成したトークン `"MTAwMToxNzAwMDAwMDAwMDAw"` → デコード `"1001:1700000000000"` → split → `"1001"` も動くため、双方向でたまたま動いてしまう状態。

> **採点ポイント**：「偶然動く」の仕組みを説明できているか。「動くからいい」では △。

## 設問 3：`utils.js` のトークン関連関数の扱い

`utils.js` の `generateToken` と `getUserIdFromToken` は現状 `app.js` から **呼ばれていない**（デッドコード）。

確認手順：
1. `app.js` 全体を `generateToken` で検索 → ヒットなし
2. `app.js` 全体を `getUserIdFromToken` で検索 → ヒットなし
3. 他のファイルでも使われていないことを確認

→ **削除して問題ない**が、削除前に以下を確認する：
- 将来の実装として意図的に残されている可能性（コメント等があるか）
- 他のブランチや別ファイルで使っていないか

削除しない場合のリスク：将来誰かが「utils.js に関数がある」と思って使い始め、`app.js` 形式とのずれが発覚しにくくなる。

> **採点ポイント**：「デッドコードである」と判断し、その確認方法と削除の判断根拠まで説明できているかを見る。

## 設問 4：どちらのトークン実装を採用すべきか

**どちらも採用すべきではない**（セキュリティ上の問題）。

両実装ともに、ユーザーIDを Base64 エンコードしているだけで、署名もない。

- 攻撃者は任意のユーザーIDを Base64 エンコードすることで、他ユーザーになりすましたトークンを自作できる
- `Buffer.from('1001').toString('base64')` = `"MTAwMQ=="` を送ると他人のアカウントに侵入可能

本来採用すべき方式：
- **JWT（JSON Web Token）**：署名付きトークン（`jsonwebtoken` パッケージ等）
- または **セッション ID 方式**：ランダムな文字列を DB/メモリに保管して照合

> **採点ポイント**：トークン実装の選択だけでなく、現状がセキュリティ上問題であることを説明できているかを見る。これを言及できれば加点。
