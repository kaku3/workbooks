# M-02: トークンの実装が 2 種類ある — 模範解答

## 設問 1：401 が返る原因

> **採点の観点**：2 種類の実装の差異を具体的に説明し、なぜ一部の API で 401 になるかの流れを追えているかを見る。

トークンの生成と解析に 2 種類の実装が混在しており、形式が互換していない。

**`app.js` の実装**（ログイン API 内）：
```js
const token = Buffer.from(String(user.id)).toString('base64')
// 例：id=1001 → "MTAwMQ=="
```

**`utils.js` の実装**（`generateToken` 関数）：
```js
function generateToken(userId) {
  return Buffer.from(userId + ':' + Date.now()).toString('base64')
}
// 例：id=1001 → "MTAwMToxNzAwMDAwMDAwMDAw"
```

`getUserIdFromToken` は `:` で split してから `[0]` を取るため、`app.js` 形式のトークンには「たまたま」動く（`:` がないため split しても `[0]` が ID になる）。

問題は `generateToken` が現状 `app.js` から**呼ばれていない**（デッドコード）ことで、将来誰かが `generateToken` を使い始めると `getUserIdFromToken` との組み合わせが変わり、既存の `app.js` 形式トークンを持つユーザーで 401 が返るようになる。

現状の 401 の直接原因は「`app.js` のログイン処理でトークンを生成し、認証チェックで `getUserIdFromToken` を使っているが、一部の API では認証チェック自体がない（A-01 の問題とも連動）」こと。

> トレース例：ログイン → `app.js` がトークン生成 → 認証ありの API は `getUserIdFromToken` で解析可能 → 認証なしの API は 401 ではなく素通り → 「一部で 401」の原因は別の API でトークンの扱いが食い違っている箇所の特定が必要。

**採点基準**
- ◎：2 実装の差異を具体的に示し、互換性の問題と「たまたま動く理由」を説明できている
- ○：2 実装が違うことを説明し、それが 401 の原因につながると説明できている
- △：「2 種類ある」と言っているが、どう違うかの具体的な説明がない
- ×：原因を特定できていない

## 設問 2：この状況をどう整理するか

> **採点の観点**：どちらを残すか・デッドコードの扱いを含めて整理できているかを見る。

**どちらも採用すべきではない**（セキュリティ上の問題）。

両実装ともに Base64 エンコードのみで署名がない。攻撃者が任意のユーザー ID を Base64 エンコードすれば他ユーザーになりすませる。

本来採用すべき方式：
- **JWT（JSON Web Token）**：署名付きトークン（`jsonwebtoken` パッケージ等）
- または**セッション ID 方式**：ランダム文字列を DB/メモリで管理して照合

**デッドコードの扱い**：

`utils.js` の `generateToken` / `getUserIdFromToken` は `app.js` から呼ばれていない。  
削除前に以下を確認する：
- 他のファイルやブランチで使われていないか
- 将来の実装として意図的に残されているか（コメント等があるか）

確認のうえ削除が推奨。残すと将来誰かが使い始めて混乱の原因になる。

**採点基準**
- ◎：「どちらも問題あり」という判断 + セキュリティの根本問題 + デッドコードの扱いをセットで説明できている
- ○：どちらを残すかの判断と理由がある
- △：「どちらかに統一する」だけで選択理由やセキュリティ問題に触れていない
- ×：「現状のままでよい」または整理方針がない

## 設問 3：なぜ「実装が 2 系統ある」状態が生まれるか・今後どう防ぐか

> **採点の観点**：構造的な原因と、再発防止の具体的な方法を説明できるかを見る。

**生まれる原因**
- 「認証の実装方法」についてチームで合意がなく、別々の人が別々の方法で実装した
- 既存コードを確認せずに新しく書いた（`app.js` に実装がすでにあることを知らなかった）
- `utils.js` に「共通処理として書いておいた」が、実際には別の実装がすでに動いていた

**今後の防止策**
- 認証の実装方法をチームで決定し、コーディング規約に明記する
- `utils.js` に追加する際は「既存の実装と重複していないか」を確認するレビュー文化を作る
- TypeScript 等で型定義を統一することで「どの関数を使うか」を自然に強制できる

**採点基準**
- ◎：原因の説明と具体的な防止策の両方がある
- ○：原因と防止策どちらもあるが一方が薄い
- △：「確認しなかったから」など表面的な原因のみ、または防止策がない
- ×：原因も防止策も説明できていない
