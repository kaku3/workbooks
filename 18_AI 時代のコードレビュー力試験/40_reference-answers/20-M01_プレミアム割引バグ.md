# M-01: プレミアム会員の割引が効かない — 模範解答

## 設問 1：問題の正確な説明

`db/users.json` を確認すると、ユーザーのフィールド名が統一されていない。

| ユーザー | 実際のフィールド名 | 値 |
|---------|------------------|-----|
| 山田太郎 (id:1001) | `type` | `"normal"` |
| 佐藤花子 (id:1002) | `memberType` | `"premium"` |
| 鈴木一郎 (id:1003) | `type` | `"normal"` |

`public/cart.html` のプレミアム判定コードは：

```js
if (user.type === 'premium') { ... }
```

佐藤花子は `user.type` が `undefined` なので、この条件は `false` となり割引が適用されない。

→ **カート画面では佐藤花子にプレミアム割引が適用されない**

なお `app.js`（サーバー側）は：

```js
if (user.memberType === 'premium') { ... }
```

こちらは佐藤花子に対しては正しく動作する。

→ **カート表示とサーバー側の注文金額が矛盾する**（カートでは通常価格、注文後はプレミアム価格）

> **採点ポイント**：「誰が」「どの画面で」影響を受けるか、カートとサーバーで挙動が違うことまで言及できているか。「なんかおかしい」では △。

## 設問 2：サーバーサイドのログで確認できること

- **注文 API（`POST /api/orders`）のレスポンス**を見ると、`app.js` 内の計算ではプレミアム割引が適用された金額になっている
- **カート画面の表示**では割引なし金額が表示されている
- この「表示金額」と「注文金額」の乖離が、バグの証拠として確認できる

具体的な確認手順：
1. 佐藤花子でログインし、商品をカートに入れる
2. カート画面に表示される合計金額（割引なし）を記録する
3. 注文を完了して `/api/orders/:id` のレスポンスを確認する
4. 注文金額がカート表示より低ければ、サーバー側でプレミアム割引が適用されていることが確認できる

> **採点ポイント**：「どうやって確認するか」の具体的な手順を書けているかを見る。

## 設問 3：最小変更での修正案

`public/cart.html` の判定を `mypage.html` と同じ方式に修正する：

```js
// 修正前
if (user.type === 'premium') {

// 修正後
if (user.type === 'premium' || user.memberType === 'premium') {
```

`mypage.html` はすでに両フィールドに対応しているため、この方式が現状への最小変更として最も安全。

ただし本来の根本修正は：
1. `users.json` のフィールド名を統一する（例：全員 `memberType` に統一）
2. すべての参照箇所を統一された名前に修正する

→ 根本修正はデータ修正を伴うため、既存注文データへの影響・他のコードへの影響確認が必要。

> **採点ポイント**：「修正案」だけでなく「なぜそれを選んだか」のトレードオフが説明できているかを見る。

## 設問 4：なぜフィールド名が揃っていないか

- `users.json` の最初の設計で `type` と決めたが、途中で「より明確な名前にしよう」と変えた
- 変更したのはサンプルデータの一部（佐藤花子だけ）に留まり、他のユーザーはそのまま
- 参照箇所のすべてを確認・修正するレビューがなかった
- または、複数の人がそれぞれ別々にデータ定義を書いた

この問題の本質は「データの整合性がコードでも JSON でも保証されていない」こと。TypeScript の型定義や JSON Schema、バリデーションを導入することで防げる。
