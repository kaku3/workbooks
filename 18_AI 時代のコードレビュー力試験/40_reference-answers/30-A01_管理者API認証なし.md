# A-01: 管理者 API に認証がない — 模範解答

## 設問 1：脆弱性の列挙と優先度評価

### 脆弱性1：管理者 API に認証が一切ない（Critical）

```js
app.get('/api/admin/orders', (req, res) => { ... })      // 認証なし
app.post('/api/admin/products', (req, res) => { ... })   // 認証なし
```

エンドポイント URL が推測されれば、ログインなしに全注文取得・商品追加が可能。  
**影響**：全ユーザーの注文情報漏洩、商品データの改ざん・追加。

### 脆弱性2：注文詳細 API に認可チェックがない（High）

```js
app.get('/api/orders/:id', (req, res) => {
  const order = orders.find(o => o.id === parseInt(req.params.id))
  // ← ログインしているユーザーの注文かどうかを確認していない
  res.json(order)
})
```

認証はされていても、他人の注文 ID を指定することで他ユーザーの注文情報が取得できる（水平権限昇格）。

### 脆弱性3：トークンが偽造可能（Critical / 別設問で既出）

Base64 エンコードのみで署名がないため、任意ユーザーの ID でトークンを自作できる。これが解決されない限り、認証の有無は意味を持たない。

### 優先度まとめ

| 優先度 | 脆弱性 | 理由 |
|--------|--------|------|
| Critical | 管理者 API 認証なし | 認証不要で全データアクセス可能 |
| Critical | トークン偽造可能 | すべての認証が無効になる |
| High | 注文への認可なし | 他人のデータが見える（水平権限昇格） |

> **採点ポイント**：3つの問題を列挙・優先度付けできているかを見る。「認証がない」だけでなく「認可がない」問題まで言及できれば ◎。

## 設問 2：修正の優先順位と進め方

**「報告してから実装する」が正解**。

理由：
- セキュリティ脆弱性はビジネスへの影響範囲が大きく、修正方針はステークホルダーの判断が必要
- 「どこまでを今回の修正範囲にするか」「いつリリースするか」は技術的判断だけで決めてはいけない
- 修正中に別の問題が発覚した場合、先に報告しておく方がリスク管理しやすい

進め方の例：
1. 脆弱性の一覧・影響範囲・緊急度を文書化してチームリーダー・PM に共有
2. 修正スコープを合意する（今回は管理者 API の認証のみ、等）
3. ホットフィックスとして優先対応

> **採点ポイント**：「まず報告」という判断とその理由が説明できているかを見る。「即修正する」は判断としてあり得るが、理由を求める。

## 設問 3：管理者 API 認証の修正案

最小構成の修正例：

```js
// 認証チェックミドルウェア
function requireAdmin(req, res, next) {
  const token = req.headers['authorization']?.replace('Bearer ', '')
  if (!token) return res.status(401).json({ error: '認証が必要です' })
  
  const userId = getUserIdFromToken(token)  // 現状の実装（要改善）
  const user = users.find(u => String(u.id) === String(userId))
  
  if (!user || user.role !== 'admin') {
    return res.status(403).json({ error: '管理者権限が必要です' })
  }
  next()
}

// 管理者 API に適用
app.get('/api/admin/orders', requireAdmin, (req, res) => { ... })
app.post('/api/admin/products', requireAdmin, (req, res) => { ... })
```

注意：
- `users.json` に `role: "admin"` フィールドの追加が必要
- トークン自体の偽造問題（M-02）が解決されていなければこの修正は不十分

> **採点ポイント**：ミドルウェアとして切り出す設計、HTTP 403 Forbidden の使用、依存する問題（トークン偽造）への言及が揃っていれば ◎。

## 設問 4：注文詳細 API の認可チェック修正案

```js
app.get('/api/orders/:id', (req, res) => {
  const token = req.headers['authorization']?.replace('Bearer ', '')
  if (!token) return res.status(401).json({ error: '認証が必要です' })
  
  const userId = getUserIdFromToken(token)
  const orders = JSON.parse(fs.readFileSync('./db/orders.json'))
  const order = orders.find(o => o.id === parseInt(req.params.id))
  
  if (!order) return res.status(404).json({ error: '注文が見つかりません' })
  
  // 認可チェック：自分の注文かどうかを確認
  if (String(order.userId) !== String(userId)) {
    return res.status(403).json({ error: 'アクセス権限がありません' })
  }
  
  res.json(order)
})
```

ポイント：
- 404（注文が存在しない）と 403（存在するが権限なし）を使い分けることで、注文 ID の存在を攻撃者に教えないようにするには、どちらも 404 を返す設計もある（セキュリティ要件次第）
- 管理者は他人の注文も参照できる必要があるため、管理者ロール確認を組み合わせる
