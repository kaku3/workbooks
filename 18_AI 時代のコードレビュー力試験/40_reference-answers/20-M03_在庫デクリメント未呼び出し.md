# M-03: 在庫を減らしていない — 模範解答

## 設問 1：在庫が減少しない仕組みの説明

`app.js` の `POST /api/orders`（注文確定 API）を確認すると、在庫減算の処理が存在しない。

```js
app.post('/api/orders', (req, res) => {
  // 注文データを作成
  // orders.json に保存
  // 在庫を減らす処理 ← ここがない
})
```

一方、在庫を減らすための `POST /api/stock/:productId/decrement` エンドポイントは存在するが、`/api/orders` の処理からそのエンドポイントを呼び出す、または直接 `products.json` の在庫を更新する処理がない。

→ 注文が何件完了しても `products.json` の在庫数は変わらない。

> **採点ポイント**：「デクリメントが呼ばれていない」という事実と「それがどのファイルのどの箇所か」を正確に指摘できているかを見る。

## 設問 2：発生しうる問題（影響範囲）

1. **オーバーセル**：在庫が 0 になっても新規注文が受け付けられる。在庫が 1 個の商品に 100 件の注文が入る可能性がある
2. **在庫表示の不整合**：商品一覧ページの在庫数が実際の手持ち在庫と乖離する
3. **注文のキャンセル対応が複雑化**：注文時に在庫を減らしていないため、「在庫に戻す」処理も実装されていないはず → キャンセル時の在庫管理が煩雑になる
4. **既存注文分の不整合**：現時点ですでに注文が入っている場合、その分の在庫は減っていないため、現在の在庫数は「実態より多い」状態になっている可能性がある

> **採点ポイント**：「在庫が減らない」だけでなく、オーバーセルや既存注文分のズレまで言及できているかを見る。

## 設問 3：修正方針と注意点

**修正方針**：`POST /api/orders` の処理内で、注文した商品の在庫を確認・減算する。

実装案（修正箇所）：
```js
app.post('/api/orders', (req, res) => {
  const products = JSON.parse(fs.readFileSync('./db/products.json'))
  const orderedProductId = req.body.productId
  const product = products.find(p => p.id === orderedProductId)

  // 1. 在庫チェック
  if (product.stock <= 0) {
    return res.status(400).json({ error: '在庫がありません' })
  }

  // 2. 注文データ作成・保存
  // ...（既存処理）

  // 3. 在庫デクリメント
  product.stock -= 1
  fs.writeFileSync('./db/products.json', JSON.stringify(products, null, 2))
})
```

**注意点**：
- 在庫チェックと在庫減算の間に別の注文が来ると「競合」が発生する（ファイルベース DB の限界）
- 既存注文分の在庫ズレをどう修正するかを事前に確認・対処が必要

> **採点ポイント**：「デクリメントを呼び出す」という修正だけでなく、在庫チェックの追加と既存データのズレについても言及できているかを見る。

## 設問 4：`/api/stock/:productId/decrement` を呼び出すか、直接書くか

**どちらでも機能的には同じだが、直接書く方が適切**。

`/api/stock/:productId/decrement` は HTTP エンドポイントなので、内部から呼ぶには `http.request` 等が必要になり、不必要に複雑になる。また外部から自由に呼び出せるエンドポイントであれば、認証なしに在庫を無限に減らせるセキュリティリスクになっている（別問題）。

整理案：
- `POST /api/orders` 内で直接在庫を減らす（ファイル操作）
- `POST /api/stock/:productId/decrement` は管理者専用の補正 API として残すか、削除する

いずれにしても「このエンドポイントが存在する理由・用途を確認してから修正する」ことが重要。
