# A-03: `app.js` をどう分割するか — 模範解答

## 設問 1：気になること・依頼者への伝え方

> **採点の観点**：「3 機能をそのまま追加することへの懸念」を責務の観点から説明し、依頼者が動けるフィードバックを出せているかを見る。  
> 単に「大きすぎる」ではなく、「このまま追加するとどう困るか」を説明できているかが問われる。

`app.js` はすでに認証・ユーザー管理・商品管理・カート・注文・在庫・クーポン・管理者操作・DB 操作が 1 ファイルに同居している。ここにさらに 3 機能を追加すると：

- ファイルがさらに巨大化し、どこに何があるかを把握するコストが上がる
- 機能間の依存関係が見えにくくなり、変更時の影響範囲の特定が難しくなる
- 現状すでに A-01（認証なし）・M-03（在庫デクリメント未呼び出し）等の問題を抱えており、その修正と新機能追加を同一ファイルで行うとリグレッションリスクが高い

**伝え方の例**

> 「3 機能の追加は技術的には可能ですが、現在の `app.js` はすでに多くの責務を持っており、このまま追加すると変更のたびに影響範囲の把握が難しくなります。  
> 機能追加と合わせてファイル分割の設計整理を先行させたいのですが、少し時間をもらえますか？  
> 分割方針の案を出して相談させてください。」

**採点基準**
- ◎：「このまま追加すると何が起きるか」を具体的に説明し、先に設計整理を提案する伝え方ができている
- ○：懸念点を示し、依頼者に提案できている
- △：「分割すべき」とは言っているが、なぜ・どう伝えるかが書かれていない
- ×：そのまま 3 機能を追加し始めている

## 設問 2：`app.js` の分割案

> **採点の観点**：`routes` と `services` / `middleware` を分けているかを見る。  
> 「どこに何を入れるか」の基準が説明できているかも問う。

```
/
├── app.js              # Express 設定・ミドルウェア登録のみ（エントリーポイント）
├── routes/
│   ├── auth.js         # POST /api/login, POST /api/logout
│   ├── users.js        # GET /api/users/:id
│   ├── products.js     # GET /api/products, POST /api/admin/products
│   ├── cart.js         # GET /POST /DELETE /api/cart
│   ├── orders.js       # GET /POST /api/orders, GET /api/orders/:id
│   ├── stock.js        # POST /api/stock/:id/decrement
│   ├── coupons.js      # GET /api/coupons/:code
│   └── admin.js        # GET /api/admin/orders
├── services/
│   ├── authService.js  # トークン生成・検証のビジネスロジック
│   ├── orderService.js # 注文作成・在庫確認のビジネスロジック
│   └── couponService.js# クーポン検証・計算のビジネスロジック
└── middleware/
    ├── authenticate.js # 認証チェック
    └── requireAdmin.js # 管理者ロールチェック
```

分割の基準：
- `routes/`：「どの URL に何をするか」のマッピングのみ
- `services/`：ビジネスロジック（計算・検証・DB 操作）
- `middleware/`：横断的な関心事（認証・認可）

**採点基準**
- ◎：routes / services / middleware の 3 層を分けており、「どこに何を入れるか」の基準を説明できている
- ○：routes と middleware は分けているが services が曖昧
- △：「routes だけ分ける」または「ファイルを分けるだけ」で責務の分け方が説明できていない
- ×：分割案を示せていない

## 設問 3：分割の進め方とリスク管理

> **採点の観点**：「どこから始めるか」の判断と、サービスを止めないための具体的な方法を説明できるかを見る。

**どこから始めるか**

認証・認可のミドルウェアから始めるのが最優先。

理由：
- 現状、認証チェックが各ルートに重複しているか存在しない（A-01 の問題）
- ミドルウェアを先に整備すると、以降のルート実装に一貫して適用できる
- 認証の問題（トークン偽造問題）の修正と合わせて対応できる

分割の順序例：
1. `middleware/authenticate.js` を作成（認証チェックの共通化）
2. `services/authService.js` を作成（トークン生成・検証を集約）
3. `routes/auth.js` を作成してルートを移動
4. 残りのルートを 1 ファイルずつ順次移動

**リスク管理**

- 1 ファイルずつ移動してコミットする（小さいコミット粒度）
- 分割前に API の手動テストケースを整理し、分割後に同じテストを実行してリグレッションを確認する
- `require` パスの変更を見落とすと実行時エラーになるため、移動後は必ず起動確認する

**採点基準**
- ◎：「なぜ認証から始めるか」の理由と、小さいコミット粒度・テストによるリグレッション確認を説明できている
- ○：始める箇所の判断と理由があり、リスク管理の方法も示せている
- △：「どこから始めるか」の判断はあるが理由がない / リスク管理に言及がない
- ×：「全部一度に分割する」または順序の判断がない

## 設問 4：`utils.js` の `readDB` / `writeDB` の整理

> **採点の観点**：デッドコードかどうかを確認したうえで判断できているかを見る。「削除すればいい」だけでは不十分。

`app.js` を確認すると、各ルートで `fs.readFileSync` / `fs.writeFileSync` を直接使っており、`utils.js` の `readDB` / `writeDB` は `app.js` から呼ばれていない可能性が高い。

**整理前にやるべきこと**

1. `readDB` / `writeDB` が `app.js` 以外のファイル（他のブランチ含む）で使われていないかを確認する
2. 将来の実装として意図的に残されているかを確認する（コメント等があるか）

**整理方針**

- 使われていないことが確認できれば削除する（デッドコードの存続は混乱の元）
- 使われているなら `app.js` の `fs` 直接アクセスと統一する。`readDB` / `writeDB` に集約するか、引き続き直接アクセスするかを決めて一方に統一する

**採点基準**
- ◎：「確認してから判断する」というアプローチと、確認後の整理方針（統一か削除か）を説明できている
- ○：確認の必要性に言及し、整理方針がある
- △：「削除すればいい」だけで確認プロセスに言及がない
- ×：`readDB` / `writeDB` について言及できていない
