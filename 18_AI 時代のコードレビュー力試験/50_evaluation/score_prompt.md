# 採点プロンプト

## 使い方

以下の手順で AI 採点（ドラフト）を行ってください。

1. このファイルの「採点プロンプト本文」をコピーして AI に貼り付ける
2. `[難易度]`・`[設問番号]` を記入する
3. `=== 問題文 ===` に `20_questions/` の該当設問を貼り付ける
4. `=== 参照コード ===` に `10_sample-ec/` の関連ファイルを貼り付ける
5. `=== 模範解答 ===` に `40_reference-answers/` の該当設問を貼り付ける
6. `=== 受験者の回答 ===` に受験者の回答を貼り付ける
7. AI の出力を「採点ドラフト」として保存し、人がレビューして最終評価を決める

## 採点プロンプト本文

```
あなたはソフトウェア設計の採点官です。
以下の情報をもとに、受験者の回答を採点してください。

難易度: [初級 / 中級 / 上級]
設問番号: [B-01 / B-02 / B-03 / M-01 / M-02 / M-03 / A-01 / A-02 / A-03]

=== 問題文 ===
（ここに問題文を貼り付け）

=== 参照コード ===
（ここに関連ファイルのコードを貼り付け）

=== 模範解答 ===
（ここに模範解答を貼り付け）

=== 受験者の回答 ===
（ここに受験者の回答を貼り付け）

## 採点基準

### A. 事実確認（コードを正しく読めているか）
- コードの動作・構造を正確に把握しているか
- 見落としや誤読がないか
- 複数ファイルの関係性を正確に追えているか

### B. 問題の言語化（「何がおかしいか」を説明できているか）
- 「動くけどおかしい」理由を自分の言葉で説明できているか
- 単なる「これは良くない」ではなく、「なぜ良くないか・何が起きるか」まで書けているか
- 教科書の丸写しでなく、このコードの文脈に合わせた説明になっているか

### C. 影響範囲の把握（変更の波及先を正確に挙げられているか）
- 修正時に影響するファイル・箇所を漏れなく挙げているか
- 見落としがある場合、それが重大な影響を持つものかどうか

### D. 改善提案の現実性（実際に実施可能な提案か）
- 理想論だけでなく、現存コードへの現実的なアプローチになっているか
- コストやリスクの感覚が適切か
- 「全部書き直す」以外の選択肢を考えられているか

### E. 判断の根拠（なぜそう判断したかを書けているか）
- AI が出した結論をそのまま書いているだけでなく、自分の言葉で根拠を書けているか
- 「自分がそう判断した理由」が読めるか

## 出力形式

以下の形式で出力してください。

### 採点結果

| 項目 | 評価 | コメント |
|------|------|---------|
| A. 事実確認 | ◎ / ○ / △ / × | |
| B. 問題の言語化 | ◎ / ○ / △ / × | |
| C. 影響範囲の把握 | ◎ / ○ / △ / × | |
| D. 改善提案の現実性 | ◎ / ○ / △ / × | |
| E. 判断の根拠 | ◎ / ○ / △ / × | |

◎ = 模範解答と同等以上 / ○ = 概ね正しい・重要な指摘あり / △ = 部分的に正しいが重要な見落としあり / × = 誤りまたは的外れ

### 重要な指摘の有無
- 模範解答で「ここは必ず指摘してほしい」とされた点が含まれているか（○ / ×）をリストアップ

### 的外れな指摘
- 存在しない問題を指摘している箇所があれば列挙

### 総評（2〜3文）
受験者のレベル感・強み・弱みを簡潔に記載してください。
```

## 評価ガイドライン（採点者向け）

### 難易度別の重点ポイント

**初級（B-01〜03）**
- A「事実確認」と B「言語化」を重視
- コードを正しく読めているか、問題を自分の言葉で説明できるかが主眼
- 改善提案は「方向性が合っていれば」十分。完璧な設計案は不要

**中級（M-01〜03）**
- C「影響範囲の把握」を重視
- 「どのファイルが影響するか」「なぜそのファイルが壊れるか」を説明できるかが主眼
- 単一ファイルの分析で終わっている回答は C が △

**上級（A-01〜03）**
- D「改善提案の現実性」と E「判断の根拠」を重視
- 「何が問題か」の指摘ではなく、「どの順序でどう対処するか・なぜその判断か」が問われる
- 「全部直す」「セキュリティリスクなので対応不要はない」だけでは不十分

### AI 生成コピーの見分け方

以下のパターンは「AI の出力をそのまま提出した可能性が高い」と判断してください：
- 設問の問い方に対して過剰に一般的な説明（このコードに言及していない）
- 模範解答と表現がほぼ同一（参照コードを全部渡した可能性）
- 「〇〇の原則に従って」「クリーンアーキテクチャでは」など教科書的キーワードのみで構成
- 設問で挙げていないファイルや関数への言及がない（コードを実際には読んでいない）

E「判断の根拠」で △ 以下の場合、採点者が直接確認することを推奨します。
