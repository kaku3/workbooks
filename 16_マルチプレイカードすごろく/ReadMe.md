# 爆弾魔 vs 解除班

ババ抜き × すごろく × 人狼の要素を組み合わせた、ブラウザで遊べる 4人用マルチプレイカードゲームです。

🎮 **[今すぐ遊ぶ](https://kaku3.github.io/workbooks/bomber-vs-defuser/index.html)**

---

## ゲーム概要

| 項目 | 内容 |
|------|------|
| プレイヤー数 | 4人（P2P通信、サーバー不要） |
| 役割 | 爆弾魔チーム × 解除班チーム（ランダム割り当て）|
| 勝利条件（爆弾魔） | パーツ A・B・C を揃えてタワーで起爆 |
| 勝利条件（解除班） | キット X・Y・Z を揃えて爆弾魔と同じマスで確保宣言 |
| 誰が爆弾魔かは秘密 | 役職はゲーム開始時にのみ本人に通知 |

毎ターン山札からカードを引き、移動カードでボード上を進みながらロケーション効果を活用。アクションカードで他プレイヤーの手札を乱しつつ、アイテムを集める。

---

## 遊び方（手順）

1. ホストが「ルームを作成」し、ルームIDを他のプレイヤーに共有
2. 残り3人がルームIDを入力して参加
3. 4人揃ったらホストが「ゲーム開始」
4. 各ターン：**ドロー → カードをプレイ（移動 or アクション）→ 次のプレイヤーへ**

---

## 技術スタック

- **フロントエンド** : 素の HTML / CSS / JavaScript（ES Modules）
- **通信** : [PeerJS](https://peerjs.com/)（WebRTC P2P）— サーバー不要
- **デプロイ** : GitHub Pages

## AI との共同開発について

このゲームは GitHub Copilot（Claude Sonnet）との会話ベースで設計・実装しました。
その経験から見えてきたことを書き留めておきます。

### アイデア出しは散漫になりやすい、でも「壁打ち」で収束する

ゼロからゲームシステムを考えるとき、最初の発想は「こういうの面白そう」という断片の集合です。
一人で考えていると散漫なまま止まりがちですが、AI に問いかけながら整理していくと
「それ、既存のメカニクスと干渉しない？」「勝利条件が複数あると情報隠匿が生きる」といった
フィードバックが即座に返ってきます。
**軸となる面白さが固まるまで、AI は優秀な壁打ち相手になります。**

### カードゲームは特殊処理の塊——実装コストが高い

カードゲームの実装は「基本ルールは単純、でも例外だらけ」が宿命です。
「このカードは通常効果の後にXXXのとき限りYYYする」——このような組み合わせ爆発を
ロジックとして正確に書き下すのは手間がかかります。
AI は仕様を日本語で伝えると対応するコードを生成してくれるため、
特殊処理の実装サイクルを大幅に短縮できました。

### バランス調整——ただし「気持ちいい」は人間にしかわからない

「パーツ工場の選択肢が多すぎてゲームが早く終わる」
「キット3種集めるより高速に動く爆弾魔が有利すぎる」
といったバランス課題は、数値として伝えれば AI が調整案を提示してくれます。

ただし、**「なんかこれ気持ちいい」「このカード使ったとき爽快感がある」という感覚は、AI にはまだ判断できません。**
面白いと感じる機能がない、というより、プレイ体験の主観的な手触りを持てないのです。
最後の一段階——数値が揃ったゲームを「気持ちよく遊べる状態」に仕上げるのは、人間の仕事として残ります。

### まとめ：「面白いを考える」ことに集中できる

- **アイデア出しの発散** → AI と壁打ちして収束させる
- **特殊ルールの実装** → 仕様を日本語で伝えてコードに落とす
- **バランス調整** → 体感をフィードバックして数値と挙動を調整する
- **最後の味付け** → 「気持ちいい」かどうかは自分で判断する

いままでは「これは面白い」と思っても、実装の物量をこなすうちにモチベーションを消費しきっていました。
AI と組むことで、退屈な肉付け作業のほとんどを短縮できます。
その分、**「どう面白くするか」を考えることと、「実際に面白いか」を確かめることに集中できる**——それが一番の変化でした。

「こういうゲームが作りたい」という核心さえあれば、あとは一緒に育てていける。そんな開発体験でした。
