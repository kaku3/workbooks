## S-03: 設計が変わったとき、図はどうなるか

**状況**

S-01 で実装した注文キャンセル機能に、新しい要件が追加されることになった。

```
「キャンセルした注文はキャンセル履歴テーブルに記録してほしい。
 あとで管理者が一覧で確認できるようにしたい。」
```

**Step 1**

AI に実装を頼んでみよう。

```
注文キャンセル時に、キャンセル履歴を記録する機能を追加したい。
キャンセル履歴には注文ID・ユーザーID・キャンセル日時・理由を保持する。
管理者が一覧で確認できる API も追加してほしい。
```

**Step 2**

S-02 で作った PlantUML の図を、今回の変更に合わせて更新してみよう。  
AI に頼んでもいいし、自分で書き足してもいい。

**設問**

1. コードの変更と図の更新、どちらが先になったか。またはどちらが難しかったか。
2. 図を更新することで、何か気づいたことはあるか（実装し忘れ・設計の矛盾など）。
3. 「コードと図が食い違っていたら、どちらが正しいか」——このプロジェクトではどう決めるべきか、あなたの考えを述べてほしい。

> **ヒント**：設計ドキュメントが「書いたら終わり」になってしまうプロジェクトは多いです。  
> 図をコードと一緒に Git で管理することで、変更履歴が残り、差分も確認できます。  
> PlantUML をコードと同じリポジトリに置く理由がここにあります。
