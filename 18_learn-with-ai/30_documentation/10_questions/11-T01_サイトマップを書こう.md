# 設計ドキュメント編（Design Documentation）

`sample-ec` プロジェクトを clone して、実際に動かした上で取り組んでください。

> **AI ツールの利用は自由です。** ただしすべての設問で「なぜそう判断したか」を自分の言葉で書いてください。

---

## T-01: サイトマップを書こう

**このチュートリアルについて**

あなたは今日、はじめてこのプロジェクトにアサインされた。  
コードを渡されたが、説明はほとんどない。  
さて、どこから手をつければいいか。

まず「どんなページが存在するか」を把握することが出発点になる。  
コードを読み始める前に、**実際に動かして、自分の目で確かめる**ことが大切だ。  
そしてその記録を、チームで共有できる形で残す。

---

**Step 1: まず動かしてみよう**

`sample-ec` を起動して、すべてのページを実際に触ってみよう。

```sh
cd 10_sample-ec
npm install
node app.js
```

ブラウザで `http://localhost:3000` を開き、各ページを順に辿ってほしい。  
このとき、**各ページのスクリーンショットを撮っておくこと**。  
（スクリーンショットは後のチュートリアルで使う）

> **ポイント**: 何ができて、何ができないか。どこからどこへ移動できるか。  
> まずは「使う人」として体験することが、全体像を掴む最短ルートだ。

---

**Step 2: サイトマップ用の Markdown を書こう**

インタラクティブサイトマップツール（`04_インタラクティブサイトマップ/12_src`）を使って、  
`sample-ec` のサイトマップを作成しよう。

まず、ツールの書式を確認する。

```sh
# ツールの ReadMe を確認
# 04_インタラクティブサイトマップ/12_src/ReadMe.md を読む
```

書式を確認したら、実際に Markdown ファイルを作成し、サイトマップを生成してみよう。

---

**Step 3: 生成したサイトマップを確認しよう**

生成されたサイトマップの画像を確認し、Step 1 で確認した実際のサイトと見比べよう。  
抜け漏れや、実態と異なる部分があれば修正する。

---

**設問**

1. すべてのページを見つけられたか。最初の一覧と、修正後の一覧を並べて、  
   「どこで見落とした / 何を足した」かを説明してほしい。
2. 実際に触ってみて、「ページ一覧には見えないが、存在するページや機能」は見つかったか。
3. この作業（サイトを触ってサイトマップを作る）に、どれくらい時間がかかったか。  
   そしてこれを、プロジェクトメンバー全員がそれぞれ個人メモとしてやっていたとしたら、  
   チーム全体でどのくらいの時間が無駄になると思うか。

> **この工程の意味**：  
> サイトマップを作る目的は「図を作ること」ではない。  
> 自分の頭の中に「このプロジェクトの規模感・構造」を作ることだ。  
> そしてそれを、チーム共有の資産として残すことで、次に来た人が同じコストをかけずに済む。  
> ここで作ったサイトマップは、以降のチュートリアルでも使い続ける。
