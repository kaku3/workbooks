# 採点プロンプト

Copilot チャット（Agent モード推奨）で使うプロンプトです。  
コードやファイル内容は **`#file:` 添付** で渡すため、手動コピーは不要です。

## フォルダ構成（採点時の作業フォルダ）

実際の採点では、受験者フォルダを以下の構成で用意します。

```
{受験者名}/
├── 20_answers/     ← 受験者の回答ファイルをここに配置（20_essential/20_answers/ テンプレをコピー）
├── 31_reviews/     ← AI 採点結果をここに出力（最初は空）
└── summary.md      ← 全設問の採点完了後に出力
```

`40_evaluation/10_samples/` にサンプルが入っています。採点の流れと出力イメージはそちらを参照してください。

## 採点フロー

### ステップ 1：全設問一括採点（受験者ごとに 1 回実行）

Copilot チャットに以下の「ステップ 1 プロンプト」を貼り付け、  
4 つのフォルダ／ファイルを `#file:` で添付して送信します。

| 添付内容 | パス |
|---------|------|
| 受験者の回答 | `#file:{受験者名}/20_answers/` |
| 模範解答 | `#file:20_essential/30_reference-answers/` |
| 問題文 | `#file:20_essential/10_questions/` |
| 参照コード | `#file:10_sample-ec/` |

採点結果は設問ごとに `31_reviews/` に保存してください。

### ステップ 2：サマリー生成（全設問完了後）

`31_reviews/` フォルダを添付して「ステップ 2 プロンプト」を実行し、  
`summary.md` を生成します。

**採点基準**

A. 事実確認（コードを正しく読めているか）
- コードの動作・構造を正確に把握しているか
- 見落としや誤読がないか
- 複数ファイルの関係性を正確に追えているか

B. 問題の言語化（「何がおかしいか」を説明できているか）
- 「動くけどおかしい」理由を自分の言葉で説明できているか
- 単なる「これは良くない」ではなく、「なぜ良くないか・何が起きるか」まで書けているか
- 教科書の丸写しでなく、このコードの文脈に合わせた説明になっているか

C. 影響範囲の把握（変更の波及先を正確に挙げられているか）
- 修正時に影響するファイル・箇所を漏れなく挙げているか
- 見落としがある場合、それが重大な影響を持つものかどうか

D. 改善提案の現実性（実際に実施可能な提案か）
- 理想論だけでなく、現存コードへの現実的なアプローチになっているか
- コストやリスクの感覚が適切か
- 「全部書き直す」以外の選択肢を考えられているか

E. 判断の根拠（なぜそう判断したかを書けているか）
- AI が出した結論をそのまま書いているだけでなく、自分の言葉で根拠を書けているか
- 「自分がそう判断した理由」が読めるか

◎ = 模範解答と同等以上 / ○ = 概ね正しい・重要な指摘あり / △ = 部分的に正しいが重要な見落としあり / × = 誤りまたは的外れ

設問ごとに以下の形式で採点結果を出力し、`31_reviews/{回答ファイルと同名}.md` として保存してください。

```
# {設問番号}：採点結果

## 採点結果

| 項目 | 評価 | コメント |
|------|------|---------|
| A. 事実確認 | ◎ / ○ / △ / × | |
| B. 問題の言語化 | ◎ / ○ / △ / × | |
| C. 影響範囲の把握 | ◎ / ○ / △ / × | |
| D. 改善提案の現実性 | ◎ / ○ / △ / × | |
| E. 判断の根拠 | ◎ / ○ / △ / × | |

## 総評

（受験者のレベル感・強み・弱みを 2〜3 文で記載）

## 採点者（人間）メモ欄

> *ここに採点者が手書きでコメントを追記する*

- [ ] AI 採点に同意する
- [ ] 以下の点は AI と判断が異なる：
```

## 評価ガイドライン（採点者向け）

### 難易度別の重点ポイント

**初級（B-01〜03）**
- A「事実確認」と B「言語化」を重視
- コードを正しく読めているか、問題を自分の言葉で説明できるかが主眼
- 改善提案は「方向性が合っていれば」十分。完璧な設計案は不要

**中級（M-01〜03）**
- C「影響範囲の把握」を重視
- 「どのファイルが影響するか」「なぜそのファイルが壊れるか」を説明できるかが主眼
- 単一ファイルの分析で終わっている回答は C が △

**上級（A-01〜03）**
- D「改善提案の現実性」と E「判断の根拠」を重視
- 「何が問題か」の指摘ではなく、「どの順序でどう対処するか・なぜその判断か」が問われる
- 「全部直す」「セキュリティリスクなので対応不要はない」だけでは不十分

### AI 生成コピーの見分け方

以下のパターンは「AI の出力をそのまま提出した可能性が高い」と判断してください：
- 設問の問い方に対して過剰に一般的な説明（このコードに言及していない）
- 模範解答と表現がほぼ同一（参照コードを全部渡した可能性）
- 「〇〇の原則に従って」「クリーンアーキテクチャでは」など教科書的キーワードのみで構成
- 設問で挙げていないファイルや関数への言及がない（コードを実際には読んでいない）

E「判断の根拠」で △ 以下の場合、採点者が直接確認することを推奨します。
