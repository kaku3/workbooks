# B-01: リファクタリングすべきか、否か、それが問題だ — 模範解答

## 設問 1：依頼をそのまま進める前に「話し合いたい」点

> **採点の観点**：コードを読んで「何かがおかしい」と自力で気づけるかを見る。  
> 列挙の網羅性より「なぜ話し合う必要があると判断したか」の理由を重視する。

`utils.js` はすでに以下の責務が 1 ファイルに同居しており、依頼通り追加してよい状態ではない。

- 日付フォーマット（`formatDate`、`formatDateTime`）
- 金額フォーマット・税計算（`formatPrice`、`calcTax`、`calcTaxIncluded`）
- バリデーション（`isValidEmail`、`isValidPhone`、`isValidPostalCode`）
- 認証トークンの生成・解析（`generateToken`、`getUserIdFromToken`）
- DB 読み書き（`readDB`、`writeDB`）
- ユーザー取得（`findUserById`、`findUserByEmail`）
- カート計算（`calcCartSubtotal`、`applyDiscount`）
- 注文ステータス変換（`ORDER_STATUS`、`getStatusLabel`）

「共通処理は `utils.js` に置く」というルール自体の妥当性を、依頼者と話し合う必要がある。

**採点基準**
- ◎：「utils.js が多くの責務を抱えていること」を根拠に、追加前に設計の見直しを提案している
- ○：「ファイルが大きい・責務が混在している」に気づき、話し合いが必要と判断している（列挙が部分的でも可）
- △：「問題あり」とは書いているが、コードの具体的な内容に言及がない
- ×：話し合うべき点が挙げられていない、またはそのまま追加してよいと判断している

## 設問 2：このまま追加した場合に起きる将来の問題

> **採点の観点**：「動くが良くない」だけでは不十分。具体的にどんな事態が起きるかを書けているかを見る。

- **変更影響の拡大**：`utils.js` は `app.js` や各 HTML から広く `require` / `import` されているため、1 箇所の変更が予期せぬ箇所に波及しやすい
- **テストの困難化**：メール送信（外部サービス呼び出し）と DB 操作・計算ロジックが同居すると、テスト時に依存関係のモック対象が増え、単体テストが書きにくくなる
- **責務のあいまい化**：「共通処理は utils に」という慣習がさらに強化され、今後も別の処理が追加され続ける
- **見通しの悪化**：ファイルが巨大化するほど、関係する関数を探すコストが上がり、修正ミスが増える

**採点基準**
- ◎：上記のうち複数の問題を、このコードの文脈に即して説明できている
- ○：「変更影響が大きくなる」「テストしにくい」など、具体的な問題を 1〜2 点挙げられている
- △：「良くない」「管理しにくくなる」など抽象的な指摘のみ
- ×：問題が挙げられていない

## 設問 3：レビュアーとして依頼者へのフィードバック

> **採点の観点**：「問題がある」と指摘するだけでなく、相手が動けるフィードバックになっているかを見る。  
> 分割案の細部よりも、伝え方と提案の方向性を重視する。

**伝え方の例**

> 「メール処理の追加自体は問題ありませんが、今の `utils.js` はすでに認証・DB・計算・フォーマットなど多くの処理が同居しています。  
> ここにさらに追加するよりも、先に責務ごとにファイルを分ける整理をしたうえで、新機能を適切な場所に追加する方が安全だと思います。  
> 少し時間をもらえれば分割案を出します。どうでしょうか？」

**修正案（分割のイメージ）**

```
lib/
├── format.js        # formatDate, formatDateTime, formatPrice
├── validate.js      # isValidEmail, isValidPhone, isValidPostalCode
├── auth.js          # generateToken, getUserIdFromToken
├── db.js            # readDB, writeDB
├── userRepo.js      # findUserById, findUserByEmail
├── cart.js          # calcCartSubtotal, applyDiscount
├── orderStatus.js   # ORDER_STATUS, getStatusLabel
└── mailer.js        # 新規追加するメール処理
```

補足：
- この構成はあくまで例示。`modules/`・`services/` など別のディレクトリ名でも、各ファイルに別の名前をつけても、「責務ごとに分離する」という判断があれば評価対象になる。
- `lib/` への分割（utils.js の整理）と、`app.js` をルーティング・ビジネスロジック・ミドルウェアに分割すること（A-03 の設問）は別のテーマ。両方提案しても構わないが、混同しないよう注意。
- 分割の粒度はチームの規模・方針による。重要なのは「メール処理を他の責務と同じファイルに混ぜない」という判断であり、細かい構成の正解は一つではない。

**採点基準**
- ◎：「なぜ分けるか」の根拠を示しつつ、依頼者が次のアクションを取れる形でフィードバックできている
- ○：問題点の指摘と具体的な分割案が両方ある
- △：「分割すべき」とは言っているが、分割案がなく相手が動けない / 分割案はあるが指摘の根拠がない
- ×：「このまま追加してください」または指摘のみで提案がない

## 設問 4：なぜ「何でも屋」になるか

> **採点の観点**：技術面のみ・文化面のみの場合は △。両面から説明できているかを見る。

**技術的な原因**
- `require('./utils')` と書けばどこからでも呼び出せるため、「どこに置くか」を判断しなくて済む
- 「どこに置くべきか」の設計指針（モジュール分割ルール）がないと、既存ファイルへの追加が最小抵抗経路になる
- ファイルが小さいうちは問題が見えないため、「とりあえず utils に追加」が繰り返される

**チームの習慣・文化的な原因**
- 「utils に置く」が慣習化すると、新メンバーも同じことをする（悪い先例への追従）
- コードレビューで「責務を分けてほしい」と指摘されなかった、または指摘が通らなかった
- 締め切りの圧力で「後で整理する」が積み重なる

**採点基準**
- ◎：技術面・文化面の両方を、このプロジェクトの文脈に即して説明できている
- ○：両面から説明できているが、一方が浅い
- △：どちらか一方のみ
- ×：「ルールがないから」など単一の原因のみ
