# M-03: 注文成功なのに、在庫そのまま — 模範解答

## 設問 1：在庫が減らない原因

> **採点の観点**：「どのファイルのどの箇所に、何が呼ばれていないか」を正確に指摘できるかを見る。

`app.js` の `POST /api/orders`（注文確定 API）を確認すると、在庫を減らす処理が存在しない。

```js
app.post('/api/orders', (req, res) => {
  // 注文データを作成して orders.json に保存
  // ← 在庫を減らす処理がない
})
```

一方、在庫を減らすための `POST /api/stock/:productId/decrement` エンドポイントは `app.js` に定義されているが、`/api/orders` の処理からそのエンドポイントを呼び出す箇所も、直接 `products.json` の在庫を更新する処理も存在しない。

→ 注文が何件完了しても `products.json` の在庫数は変わらない。

**採点基準**
- ◎：「`/api/orders` に在庫減算処理がない」という事実と「デクリメントエンドポイントが呼ばれていない」を具体的に指摘できている
- ○：「注文処理に在庫を減らす処理がない」を指摘できている
- △：「在庫が減らない」とは言っているがコードの根拠がない
- ×：原因を特定できていない

## 設問 2：修正アプローチと修正前の検討事項

> **採点の観点**：「デクリメントを呼べばよい」だけでなく、在庫チェック・既存データのズレ・実装方法の選択まで考えられているかを見る。

**修正方針**：`POST /api/orders` 内で注文商品の在庫を確認・減算する。

実装案：

```js
app.post('/api/orders', (req, res) => {
  const products = JSON.parse(fs.readFileSync('./db/products.json'))
  const product = products.find(p => p.id === req.body.productId)

  // 1. 在庫チェック
  if (!product || product.stock <= 0) {
    return res.status(400).json({ error: '在庫がありません' })
  }

  // 2. 注文データ作成・保存（既存処理）
  // ...

  // 3. 在庫デクリメント
  product.stock -= 1
  fs.writeFileSync('./db/products.json', JSON.stringify(products, null, 2))
})
```

**`POST /api/stock/:productId/decrement` を内部から呼ぶべきか？**  
HTTP エンドポイントを内部から呼ぶには `http.request` 等が必要で不必要に複雑になる。直接ファイルを更新する方が適切。デクリメントエンドポイントは管理者向け補正 API として残すか、削除を検討する。

**修正前に確認すべきこと**
- 既存の注文データ分の在庫ズレを把握・補正する必要があるか
- 複数注文が同時に来た場合の在庫競合（ファイル DB の限界）をどこまで許容するか

**採点基準**
- ◎：在庫チェックの追加・既存データのズレ・実装方法の選択（直接更新 vs エンドポイント呼び出し）まで言及できている
- ○：在庫チェックを含む修正案と、修正前の確認事項がある
- △：「デクリメントを呼べばよい」だけで、在庫チェックや既存データへの言及がない
- ×：修正方針がない

## 設問 3：設計上の根本原因

> **採点の観点**：「なぜこの実装になったか」を設計の観点から説明できるかを見る。

**根本原因**：注文処理と在庫管理が設計上で分離されており、「注文を確定したら在庫を減らす」という連動が保証されていない。

- エンドポイントごとに独立して実装されており、トランザクション的な「注文確定 = 在庫減算」がセットで担保される仕組みがない
- `POST /api/stock/:productId/decrement` が存在することから「在庫デクリメントは必要」という認識はあったが、「誰が呼ぶか」が決まっていなかった（フロントから呼ぶつもりだったかもしれない）
- テストや受け入れ確認で「注文後の在庫数」を確認していなかった

この種の問題（「2 つの操作がセットで実行される必要があるが、片方が抜ける」）はサービス層でまとめることで防げる。`OrderService.create()` の中で注文作成と在庫減算をセットにするなど。

**採点基準**
- ◎：「設計上の分離」が問題であることを説明し、解決の方向性（サービス層での集約など）まで言及できている
- ○：「呼ばれていなかった原因」を設計・プロセスの観点から説明できている
- △：「実装漏れ」「確認不足」など個人の問題として扱っている
- ×：原因の説明がない
