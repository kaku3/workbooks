# A-01: その実装、進めていいですか？ — 模範解答

## 設問 1：依頼を進める前に気になる点

> **採点の観点**：コードを読んで「追加前に問題がある」と自力で気づき、何が問題かを説明できるかを見る。  
> 認証の有無だけでなく、認可の問題まで把握できているかも問う。

参照として示された `GET /api/admin/orders` を確認すると、認証が一切かかっていない。

**脆弱性 1：管理者 API に認証なし（Critical）**

```js
app.get('/api/admin/orders', (req, res) => { ... })    // 認証なし
app.post('/api/admin/products', (req, res) => { ... }) // 認証なし
```

URL が推測されれば、ログインなしで全注文取得・商品追加が可能。影響：全ユーザーの注文情報漏洩、商品データの改ざん。

**脆弱性 2：注文詳細 API に認可チェックなし（High）**

```js
app.get('/api/orders/:id', (req, res) => {
  const order = orders.find(o => o.id === parseInt(req.params.id))
  // ← ログインしているユーザーの注文かどうかを確認していない
})
```

ログイン済みでも他人の注文 ID を指定すれば他ユーザーの情報が取得できる（水平権限昇格）。

**脆弱性 3：トークンが偽造可能（Critical）**

Base64 エンコードのみで署名がないため、任意ユーザー ID のトークンを自作できる。これが解決されない限り、認証を追加しても意味をなさない。

**採点基準**
- ◎：認証なし（管理者 API）・認可なし（注文詳細）・トークン偽造の 3 点を挙げ、それぞれの影響を説明できている
- ○：管理者 API の認証なしを指摘し、依頼をそのまま進めるべきでないと判断できている
- △：「認証がない」とは言っているが、どの API が対象で何が起きるかの説明がない
- ×：そのまま実装を始めようとしている

## 設問 2：セキュリティリスクとしてのレポート（優先度）

> **採点の観点**：リスクの大きさ・影響範囲・対処コストのバランスで優先度をつけられているかを見る。

| 優先度 | 脆弱性 | 理由 |
|--------|--------|------|
| Critical | 管理者 API 認証なし | 認証不要で全データアクセス・改ざん可能 |
| Critical | トークン偽造可能 | すべての認証が無効になる。認証を追加しても意味をなさない |
| High | 注文への認可なし | ログイン済みなら他人のデータが見える（水平権限昇格） |

対処コストの観点：
- 管理者 API 認証：ミドルウェアを 1 つ追加して適用するだけで最小コスト
- トークン偽造：JWT 等への変更が必要で影響範囲が広い（全認証処理の変更）
- 注文認可：各 API に `userId` チェックを追加する

→ 「管理者 API への認証追加」は対処コストが低くリスクが高いため最優先。

**採点基準**
- ◎：3 つの問題を優先度付きで列挙し、対処コストと影響範囲のバランスを説明できている
- ○：2 つ以上の問題を列挙し、優先度の根拠がある
- △：「危険だ」と言っているが、具体的な問題・優先度・根拠がない
- ×：リスクを列挙できていない

## 設問 3：「注文キャンセル機能を追加してほしい」への対応

> **採点の観点**：技術的判断だけでなく、チームや事業への影響を考えた意思決定ができているかを見る。

**「先に報告してから、修正スコープを合意したうえで実装する」が推奨**。

理由：
- セキュリティ脆弱性はビジネスへの影響範囲が大きく、修正方針はステークホルダーの判断が必要
- 「どこまでを今回の修正範囲にするか」「いつリリースするか」は技術的判断だけで決めてはいけない
- キャンセル機能の追加と脆弱性修正を同時に進めた場合、リグレッションが起きたとき原因の特定が難しくなる

進め方の例：
1. 脆弱性の一覧・影響範囲・緊急度を文書化し、PM・チームリーダーに共有
2. 修正スコープ（今回は管理者 API の認証追加のみ、等）を合意する
3. キャンセル機能は脆弱性修正後に着手

**採点基準**
- ◎：「先に報告」という判断とその理由が明確で、進め方の具体案がある
- ○：「先に報告」という判断とその理由が説明できている
- △：「報告する」だけで、なぜか・どう進めるかが書かれていない / 「即実装」だが理由がある
- ×：そのまま実装する（脆弱性を報告しない）
