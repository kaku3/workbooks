# M-01: プレミアム会員なのに、割引がゼロ — 模範解答

## 設問 1：割引が効かない原因

> **採点の観点**：「誰が・どの画面で・なぜ」影響を受けるかをコードを根拠に説明できるかを見る。  
> ただ「フィールド名が違う」だけでなく、どのコードがどう動くかの流れまで説明できているかを問う。

`db/users.json` のフィールド名が統一されていない。

| ユーザー | 実際のフィールド名 | 値 |
|---------|------------------|-----|
| 山田太郎 (id:1001) | `type` | `"normal"` |
| 佐藤花子 (id:1002) | `memberType` | `"premium"` |
| 鈴木一郎 (id:1003) | `type` | `"normal"` |

`public/cart.html` のプレミアム判定：

```js
if (user.type === 'premium') { ... }
```

佐藤花子は `user.type` が `undefined` なので条件が `false` → **カート画面で割引が表示されない**。

一方 `app.js` のサーバー側判定：

```js
if (user.memberType === 'premium') { ... }
```

こちらは佐藤花子に正しく動作する → **カート表示は通常価格、注文後の金額はプレミアム価格という矛盾が発生する**。

**採点基準**
- ◎：フィールド名の不一致・カートとサーバーで挙動が違うこと・矛盾が起きることまで説明できている
- ○：フィールド名の不一致と「誰の・どの画面で」割引が効かないかを説明できている
- △：「フィールド名が違う」と言っているが、どのコードがどう動くかに言及がない
- ×：原因を特定できていない

## 設問 2：修正方法（影響が最小のアプローチ）

> **採点の観点**：「修正案」だけでなく「なぜその修正を選んだか」のトレードオフを説明できているかを見る。

**短期：`cart.html` の判定を両フィールドに対応させる（最小変更）**

```js
// 修正前
if (user.type === 'premium') {

// 修正後
if (user.type === 'premium' || user.memberType === 'premium') {
```

`mypage.html` がすでに同様の対応をしているため、この方式が現状への最小変更として最も安全。

**根本修正（中長期）**：`users.json` のフィールド名を全員で統一（例：全員 `memberType`）。  
ただしデータ修正を伴うため、既存注文データへの影響と他コードの参照箇所を事前に確認する必要がある。

**採点基準**
- ◎：短期修正と根本修正を分けて説明し、それぞれのリスクと根拠を説明できている
- ○：修正案があり「なぜその修正か」の理由がある
- △：「修正案」だけで選択理由がない
- ×：修正案がない、または「users.json を全部書き直す」だけ

## 設問 3：このバグはなぜ生まれたか・同様の不整合はあるか

> **採点の観点**：根本原因を構造的に説明できるか。同様の問題を「自分で探しに行く」視点があるかを見る。

**バグが生まれた原因**
- `users.json` の設計途中でフィールド名が変更されたが、一部のデータのみ更新された（移行が不完全）
- 参照箇所を網羅的に確認・修正するレビューがなかった
- `type` と `memberType` のどちらが正しいかを決める設計指針がなかった

**同様の不整合が潜む可能性**
- `db/` 以下の他の JSON ファイル（`products.json`、`orders.json`）に同様のフィールド名の不統一がないか確認が必要
- `mypage.html` がすでに `|| user.memberType` で両対応しているのは、過去に同様の問題に遭遇した痕跡の可能性がある

**採点基準**
- ◎：根本原因の説明 + 同様の問題を探す視点（他ファイルの確認、mypage.html の対応が手がかりになること）を示せている
- ○：根本原因を複数の観点から説明できている
- △：「フィールド名を途中で変えたから」など単一原因のみ
- ×：原因を説明できていない
