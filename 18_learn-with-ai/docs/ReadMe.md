# AI 時代のコードレビュー力試験 — コンセプト・検討・仕様案

## 1. コンセプト

**「仕様書なしで増改築され続けた実在感のある EC サイト（`sample-ec`）を舞台に、追加開発の設計判断を問う」** 試験。

- 受験者は `sample-ec` プロジェクト全体を clone し、コードベースを把握する
- 各設問は「この機能を追加してほしい」という依頼の形で来る
- 受験者はプロジェクト全体を読んだうえで、**どこに・どう手を入れるか、そのリスクは何か** を回答する
- AI ツールの利用は自由。ただし問いは「AI に丸投げしても妥当な回答が返ってこない」設計にする

コーディングテストとは異なり、「書ける」ではなく **「読める・判断できる・リスクを説明できる」** 能力を測る。

### なぜ「単一プロジェクト」でないといけないか

孤立したコードスニペットを問題として出す方式には、致命的な弱点がある。

```
スニペット問題 → AI に貼り付ける → そこそこ妥当な回答が返ってくる → 試験の意味がない
```

**「影響範囲はどこか」「リファクタリングのリスクは何か」「優先度をどうつけるか」** という実務で最も重要な判断は、
プロジェクト全体のコンテキストがないと問えない。そして AI はプロジェクト全体を知らない。

`sample-ec` は：
- 複数人が・複数年にわたって・仕様書なしで継ぎ足してきた想定のコードベース
- 問題を意図的に「埋めて」ある（孤立させず、他のファイルと依存関係を持たせる）
- 設問は「この変更はどのファイルに波及するか」「なぜここは触りにくいか」を問う形にする

> AI に `sample-ec` 全体を渡して問いを解かせることは可能。だがそれ自体が「AI を使って判断する」スキルを使っている。
> AI の出力を鵜呑みにせず、自分の言葉で「なぜそう判断するか」を書ける人間を選別できる問い方をする。

### なぜ今この試験が必要か

AI コード生成ツールが普及した現在、エンジニアの役割は「コードを書く」から
**「AI が生成したコードが妥当かどうかを判断する」** へとシフトしつつある。

しかし AI は「動くコード」を生成するのは得意でも、
「プロジェクト規模・チーム・将来の変更頻度に合った設計」を自律的に選ぶことは苦手である。
その判断を担えるのは、**設計の良し悪しを嗅覚で感じ取れる人間** だけだ。

この試験が測りたいのは、教科書的な知識（「Single Responsibility Principle とは何か」を言えるか）ではない。
**手に馴染む程度に設計パターンを使い続けた結果として身についた、問題コードへの違和感を言語化する力** である。

> 「このコードは動く。でも何かおかしい」——その感覚を持ち、根拠をもって説明できるか。

### 出題意図

1. **AI 生成コードの妥当性検証力**：AI が出してきたコードをそのまま採用していいか判断できるか
2. **設計嗅覚の言語化**：「なんとなく嫌」を「〇〇の原則に反するから△△という問題が起きる」と説明できるか
3. **規模感の適切さの判断**：ガチガチの設計パターンが小規模プロジェクトにかえって害になるケースを見抜けるか
4. **改善提案の現実性**：理想論ではなく、今あるコードに対して現実的に実施できる改善を提案できるか

## 2. 提供形式

| 項目 | 内容 |
|------|------|
| 配布方法 | Git リポジトリを clone して利用 |
| 受験環境 | エディタ・ブラウザのみ（Node.js が動けば十分） |
| 回答形式 | Markdown ファイルに記述（自由記述） |
| AI 利用 | **自由**。ただし「なぜそう判断したか」の説明を必須とする |
| 採点 | 評価用プロンプトを AI に投げてドラフト採点 → 人がレビュー |

```
repository/
├── README.md                  # 試験の説明・進め方・AI 利用ルール
├── 10_sample-ec/              # ← 問題の舞台となる EC サイト本体
│   ├── package.json
│   ├── app.js                 # Express サーバー兼ビジネスロジック（意図的に肥大化）
│   ├── utils.js               # 何でも屋ファイル（認証・DB・計算・フォーマットが混在）
│   ├── db/
│   │   ├── users.json         # パスワード平文・type/memberType フィールド不一致
│   │   ├── products.json
│   │   ├── orders.json
│   │   └── cart.json
│   └── public/
│       ├── common.js          # utils.js と関数が重複
│       ├── cart.html          # 消費税・プレミアム判定が app.js と食い違い
│       ├── checkout.html      # クーポン計算がフロントのみ
│       ├── mypage.html
│       └── ...
└── 20_essential/
    ├── 10_questions/          # 設問ファイル（難易度別）
    │   ├── 10_beginner.md
    │   ├── 20_intermediate.md
    │   └── 30_advanced.md
    ├── 20_answers/            # 受験者が回答を書く場所（空テンプレート）
    ├── 30_reference-answers/  # 模範解答
    └── 40_evaluation/         # 採点用プロンプト集
```

## 3. 難易度区分の考え方

難易度は「問題の見つけやすさ」ではなく **「判断に必要なコンテキストの広さ」** で決める。

| 難易度 | コンテキストの広さ | 問いの典型 |
|--------|-----------------|----------|
| **初級** | 1 ファイル内で完結 | 「このファイルの問題を列挙して」「どう直す？」 |
| **中級** | 複数ファイルにまたがる | 「この変更を加えると他のどのファイルに影響が出る？」 |
| **上級** | プロジェクト全体 + ビジネス判断 | 「リファクタリングの優先度と進め方を提案して。コストとリスクも含めて」 |

### 初級（Beginner）
- 1 ファイルを読めば問題が見える
- 問いは「どこが問題か・どう直すか」
- AI に貼り付ければ指摘は出る。**だが「なぜそれが問題を引き起こすのか」を自分の言葉で書けるかを問う**
- 例：`utils.js` の肥大化（日付・金額・バリデーション・認証・DB・カート計算が同居）、`utils.js` と `public/common.js` の関数重複、消費税計算が `app.js`・`cart.html`・`checkout.html` の3箇所で全て異なる

### 中級（Intermediate）
- 複数ファイルにまたがる問題で、変更の影響範囲を正確に把握できるかを問う
- 「修正するとしたらどのファイルを変える？どこが壊れるリスクがある？」
- AI はファイル単体で答えられるが、**依存関係・デグレリスクはプロジェクト全体を知らないと答えられない**
- 例：プレミアム割引のフィールド名不一致（`app.js` は `memberType`、`cart.html` は `type` を参照 → 佐藤花子の割引が効かない）、トークン生成ロジックが `app.js` と `utils.js` で互換性なし、在庫デクリメントAPIが存在するが注文ルートから呼ばれていない

### 上級（Advanced）
- プロジェクト全体 + ビジネス判断が必要な問題
- 「優先順位と根拠・リファクタリング計画・ステークホルダーへの説明方針」まで問う
- **AI が出した回答を素材に、自分の判断を重ねて書く形** が実務に最も近い
- 例：管理者API（`GET /api/admin/orders`・`POST /api/admin/products`）に認証が一切ない、クーポン割引がフロントのみで計算されサーバーに送られていない、`app.js` をどう分割するか（注文キャンセル・ポイント・在庫アラート追加を題材に）

## 4. 設問の形式

各設問は **「〇〇という機能追加依頼が来た」** という実務に近い形で出す。

```markdown
## [設問タイトル]

### 依頼内容
「送料無料キャンペーン（3,000 円以上）を今週末までに追加してほしい」

### 前提
- `sample-ec` の `order.js` `cart.js` を確認してください
- テストコードはありません
- 本番稼働中のサービスです

### 問い
1. どのファイルのどの箇所に手を入れますか？
2. 影響を受ける可能性がある他の箇所はどこですか？
3. この変更でデグレが起きやすい箇所はどこですか？なぜですか？
4. あなたが「手を入れる前にやること」を挙げてください
```

孤立スニペットと異なり、`sample-ec` 全体を把握した上でないと答えられない問いになる。

## 5. 評価プロンプトの設計方針

- `20_essential/40_evaluation/score_prompt.md` に汎用採点プロンプトを用意
- 受験者の回答 + `checklist.md` + 問題コードを AI に渡して採点
- 出力形式：**指摘漏れ / 的外れな指摘 / 改善提案の質** を項目ごとに評価
- 完全自動採点ではなく「AI がドラフト → 人がレビュー」の補助ツールとして位置づける

## 6. 検討事項・TODO

### 言語方針

**JavaScript オンリー**（`sample-ec` は全て `.js`）

| 理由 | 内容 |
|------|------|
| 間口の広さ | TypeScript 環境構築なしに誰でも読める |
| AI ありきなら型より設計の方が本質 | 「型がない」こと自体も設計問題の一つになる |
| 実際の現場感 | JS + 型なし混在は現実のコードベースによくある状態 |

> TypeScript 版は将来の拡張セットとして検討。まず JS で問題の質を磨く。

### 決定が必要な事項
- [x] 対象言語：JavaScript オンリー（sample-ec 全体を JS で統一）
- [x] 問題の舞台：`10_sample-ec`（仕様書なし増改築 EC サイト）に一本化
- [x] `10_sample-ec` のスコープ：`app.js` + `utils.js` + `db/*.json` + `public/*.html` で実装済み
- [x] 問題数：難易度別 3 問ずつ（初級 B-01〜03、中級 M-01〜03、上級 A-01〜03）
- [ ] 模範解答の公開タイミング：提出後即時 or 試験期間終了後
- [ ] リポジトリの公開範囲：Public / Private

### 懸念事項
- **`sample-ec` の作り込みコストが高い**：問題が全部同じプロジェクトに乗る分、コードベース自体の品質・リアリティが問題の質を左右する
- **AI に全ファイルを投げれば解ける可能性**：context window が広いモデルだとプロジェクト丸ごと渡せる → 問いは「AI 出力を素材に自分の判断を重ねる」形に設計する
- **採点の難しさ**：正解が一意でない問い（優先度・方針）の採点精度をどう担保するか

### `sample-ec` に埋め込まれた問題パターン（実装済み）

| ファイル | 問題 |
|---------|------|
| `utils.js` | 日付・金額・バリデーション・認証・DB・ユーザー・カート計算が同居（何でも屋化） |
| `utils.js` vs `public/common.js` | `formatPrice` / `formatDate` が両ファイルに重複 |
| `utils.js` vs `app.js` | `calcTax` は `Math.round`、`app.js` は `Math.floor`（端数処理が違う） |
| `utils.js` vs `app.js` | `generateToken` はタイムスタンプ付き base64、`app.js` は ID のみ base64（互換性なし） |
| `app.js` vs `cart.html` | プレミアム判定フィールドが `memberType`（サーバー）と `type`（フロント）で不一致 |
| `db/users.json` | 佐藤花子のみ `memberType: 'premium'`、他ユーザーは `type: 'normal'`（フィールド名不統一） |
| `db/users.json` | パスワード平文保存 |
| `app.js` | `GET /api/orders/:id` に認可チェックなし（他人の注文が見える） |
| `app.js` | `GET /api/admin/orders` / `POST /api/admin/products` に認証なし |
| `app.js` | `POST /api/stock/:productId/decrement` が存在するが注文ルートから呼ばれていない |
| `checkout.html` | クーポン割引がフロントのみで計算され、サーバーに送られていない |
| `app.js` | クーポンコードが `COUPONS` 配列にハードコード（DB ファイルなし） |
| `checkout.html` | 消費税計算が `Math.floor(subtotal * 1.1) - subtotal`（3箇所目の別実装） |

## 7. AI 時代の技術負債とリファクタリング

### 大局は変わらない

AI がコードを書く時代になっても、**技術負債はたまる**。
AI は「今動くコード」を生成するが、「このプロジェクトの将来を考慮した設計」はしない。
むしろ、爆速生成できる分だけ **細かくリファクタリングしないと技術負債の蹴みが早くなる**リスクがある。
結局、人間の判断と手入れが不可欠な点は変わらない。

### リファクタリングのジレンマ

```
技術負債がたまり、修正コストが上がる
      ↓
リファクタリングしたい
      ↓
でもテストがない・影響範囲が不明・デグレが怪い
      ↓
リファクタリングできない
      ↓
更に負債がたまる
```

このループを回避するには、**小さく・安全に・継続的に** リファクタリングする人間が必要である。

### この試験が測りたい現場力

| 能力 | 具体的な場面 |
|------|------------|
| リファクタリングの必要性を見極める | 追加開発前に「ここは手を入れるべきか」を判断できる |
| **コスト見合いで回道を選べる** | デグレリスク・期限・将来の変更頻度をみて「今やる / 後回しにする / 決して踏まない」を分ける |
| 影響範囲を小さく押さえる | 複数の実装案から「現存コードへの手入れ最小」の案を選べる |
| 設計の良し悪しを説明できる | 「リファクタリングしたい」をチーム・ステークホルダーに説得できる |
